# AI 音乐小伙伴：单伙伴先行测试与人格 × 学习风格细化路线

规划日期：2026-07-18  
状态：**PLANNING ONLY / 不推进项目运行时代码**  
适用项目：`aaaycc931-droid/my-first-app`  
上位路线：`docs/ai-music-companion-agent-roadmap-2026-07-18.md`

## 1. 本次细化决策

项目不在最初阶段同时规划或制作四只伙伴。

第一阶段只制作并测试 **一只伙伴**，先验证以下核心问题：

- 用户是否愿意让角色长期陪伴学习；
- 伙伴是否能提升练习完成率、复练意愿和跨模块使用；
- “伙伴先天人格 × 用户选择的学习风格”是否能形成自然、稳定且不失真的表现；
- 伙伴成长是否能真实反映音乐学习，而不是变成与学习无关的宠物养成；
- 动画、语音、唱名和智能体能力是否会干扰音频延迟、录音与专业练习。

第一只伙伴的动物种类、名字、视觉形象、背景故事、核心人格、口头禅、完整台词库和声音暂不冻结，统一标记为 `TBD`。在这些内容确定前，允许使用内部代号：

- `companion_alpha_01`
- 中文工作名：**首发测试伙伴**

路线只先冻结角色系统的结构、边界和验收方式，不提前编造最终角色设定。

## 2. 长期方向不变：未来扩展为多伙伴

单伙伴先行只是开发与验证策略，不是最终产品范围。

未来仍计划加入多个拥有独立形象、先天人格、成长表现、动作语言和角色故事的音乐伙伴。用户选择某位伙伴陪伴后，该伙伴建立独立的陪伴关系并成长。

长期产品原则：

> 用户选择的不是一个皮肤，而是一位有稳定人格、能以不同学习方式陪伴自己成长的音乐伙伴。

所有伙伴共享可靠的音乐核心能力，但不共享完全相同的人格表现。

## 3. 伙伴人格 × 学习风格模型

### 3.1 伙伴先天人格决定“它是谁”

每位伙伴拥有相对稳定、不能被学习风格完全覆盖的核心人格，包括但不限于：

- 内向或外向；
- 冷静或热情；
- 克制或外显；
- 温柔或直接；
- 谨慎或冒险；
- 严肃或幽默；
- 说话简短或详细；
- 主动性和情绪表达的自然上下限；
- 标志性动作、措辞与价值观。

学习风格不能把高冷伙伴改造成与天生活泼伙伴相同的角色，也不能把活泼伙伴完全压平成没有情绪的系统助手。

### 3.2 学习风格决定“它怎么教”

首版预留以下学习风格：

1. **温柔讲解**：更耐心、更重视情绪安全、解释更完整；
2. **活泼互动**：更多接唱、模唱、小游戏、动作与主动邀请；
3. **精简专业**：反馈直接、数据优先、减少寒暄和动画；
4. **安静陪伴**：降低主动说话频率，主要在关键节点出现。

未来可在验证后加入“挑战教练”等风格，但首版不因追求数量增加复杂度。

### 3.3 最终表现公式

设计与实现统一遵守：

> 最终行为 = 伙伴先天人格 + 学习风格调节 + 当前练习情境 + 用户关系阶段 + 安全与专业边界。

学习风格改变的是表达强度、教学节奏和互动方式，不替换角色的核心人格。

### 3.4 同一风格在不同伙伴上的长期差异

未来多伙伴阶段，同样选择“活泼互动”时：

- 天性高冷的伙伴可以变得更主动、更愿意接唱和开轻微玩笑，但仍保持克制、话少和标志性冷静；
- 天生活泼的伙伴可以表现出更高动作幅度、更频繁的游戏邀请和更外显的庆祝；
- 两者使用相同教学策略目标，但台词、动作、主动程度、情绪强度和节奏必须保留明显差异。

因此，学习风格不是可替换人格包，而是作用于每位伙伴自身参数范围内的调节器。

## 4. 单伙伴测试阶段如何验证该体系

虽然首版只有一只伙伴，系统从第一天就要按未来多伙伴结构设计，避免以后重构。

### 4.1 首版必须区分的两层

**固定人格层**：

- 角色核心特质；
- 说话与动作的自然上下限；
- 不同情境下不会违反的角色原则；
- 标志性表达和禁用表达。

**学习风格层**：

- 主动提示频率；
- 讲解详细度；
- 鼓励强度；
- 游戏和挑战频率；
- 动画活跃度；
- 纠错直接程度；
- 下一步建议的主动程度。

即使第一位伙伴的人格尚未确定，数据结构、内容模板和测试用例也必须分开存储这两层。

### 4.2 单伙伴测试的关键目标

首发测试不是比较不同伙伴，而是确认：

- 同一位伙伴切换学习风格后，用户能明显感到教学方式变化；
- 切换风格后，用户仍然认为“它还是同一位伙伴”；
- 风格变化不会造成角色前后矛盾；
- 专业反馈事实保持一致，变化只发生在表达和教学组织上；
- 用户能随时改变学习风格，不损失伙伴成长与学习进度。

### 4.3 必须包含的组合测试

首发伙伴至少使用四种学习风格完成相同场景测试：

- 第一次唱错一个音；
- 连续三次在同一音程出错；
- 完成一次明显进步的重唱；
- 用户练习时间过长；
- 麦克风无有效输入；
- 分析置信度不足；
- 用户要求少说话；
- 用户主动请求小游戏。

每个场景都要验证：

1. 底层事实和建议目标一致；
2. 风格表达明显不同；
3. 角色核心人格没有被覆盖；
4. 没有制造羞耻、焦虑、负罪感或错误承诺。

## 5. 首发伙伴的人格设计流程

首发伙伴的具体人格不在本文件中直接决定，后续按以下顺序冻结。

### 5.1 先定义功能角色

先回答它在产品中的主要职责：

- 更偏陪练、老师、同学还是音乐搭档；
- 更适合儿童、初学者、成人还是兼容型用户；
- 在专业场景中需要保持多低的干扰；
- 它与钢琴、乐谱、视唱和歌曲模块分别如何互动。

### 5.2 再定义人格骨架

至少冻结：

- 3 个核心特质；
- 2 个轻微缺点或限制；
- 3 个价值观；
- 3 个标志性动作；
- 正常、开心、担心、认真、困惑时的表达方式；
- 它绝不会说或做的事情。

人格需要有辨识度，但不能通过攻击、嘲讽、制造依赖或故意不专业来建立特色。

### 5.3 再选择形象

动物种类和视觉形象应服务人格与音乐动作，而不是先画一个可爱角色再强行套设定。选择时需评估：

- 能否自然表现呼吸、唱歌、听音、打拍和演奏；
- 小尺寸下轮廓是否清楚；
- 儿童与成人是否都能接受；
- 是否容易制作多状态动画；
- 是否具有原创性和品牌识别；
- 是否容易与后续伙伴形成不同轮廓和性格谱系。

### 5.4 最后编写故事与台词

背景故事不应压过学习功能。故事需要解释它为什么喜欢音乐、为什么愿意陪用户成长，但不应要求用户长期阅读剧情才能理解功能。

台词库按“事实层 → 教学意图层 → 人格表达层 → 学习风格调节层”生成和审核。任何角色台词都不能改变底层分析事实。

## 6. 首发伙伴 MVP 范围

首版只验证最关键闭环，不制作完整宠物游戏。

### 6.1 必须实现

- 首页和练习页的轻量角色存在；
- 待机、倾听、思考、示范、鼓励、低置信和安静七类基础状态；
- 陪伴、专注和完全隐藏模式；
- 温柔讲解、活泼互动、精简专业、安静陪伴四种学习风格；
- 根据真实练习事件反应；
- 播放标准音、基础唱名、音阶和短范唱；
- 指向具体错误音符或片段；
- 启动重听、慢速、循环和重唱；
- 学习风格随时可改；
- 伙伴成长数据与用户音乐学习数据分离；
- 无网络时保留基础陪练能力。

### 6.2 暂不实现

- 多伙伴选择与收集；
- 完整角色故事章节；
- 大量服装、家具和房间装饰；
- 抽卡、稀有度或角色强弱；
- 复杂开放式聊天；
- 高质量生成式歌声；
- 多声部自由合唱；
- 系统级桌面悬浮宠物；
- 公开社区、伙伴排行榜或竞争养成；
- 因换风格或中断学习产生惩罚、退化或关系破裂。

## 7. 首发核心用户流程

### 7.1 首次进入

1. 用户看到首发伙伴的简短介绍；
2. 用户选择一种学习风格；
3. 系统提供四种风格的同场景预览；
4. 用户确认后进入应用；
5. 系统明确说明风格可以随时修改，小伙伴也可以完全隐藏。

首次进入不要求用户理解长期人格系统，也不使用长剧情阻塞核心功能。

### 7.2 练习闭环

1. 用户开始一段视唱或模唱；
2. 伙伴进入倾听状态，但不覆盖实时音高界面；
3. 用户提交后，底层系统完成可信分析；
4. 伙伴依据证据指出具体问题；
5. 按当前学习风格组织解释；
6. 伙伴播放标准音或短范唱；
7. 自动准备不覆盖原记录的复练片段；
8. 用户重唱并看到前后变化；
9. 伙伴成长只在真实完成或能力进步后获得记录。

### 7.3 切换学习风格

1. 用户打开伙伴设置；
2. 选择其他学习风格；
3. 预览同一伙伴的新表达方式；
4. 确认切换；
5. 学习数据、伙伴关系和成长不清零；
6. 新风格从下一次教学节点生效，避免一句话中途突然变人格。

## 8. 成长系统细化

### 8.1 用户音乐能力：账号或本机共享

以下属于用户自身，不绑定某位伙伴：

- 课程进度；
- 技能掌握度；
- 音准、节奏、听辨和识谱画像；
- 钢琴与制谱项目；
- 歌曲、录音和练习历史；
- 已完成的真实学习成就。

未来切换伙伴时，这些数据继续保留。

### 8.2 伙伴独立成长

以下属于伙伴与用户的陪伴关系：

- 陪伴阶段；
- 已解锁的动作和表情；
- 已学会的音乐互动表现；
- 与用户共同完成的代表作品；
- 专属纪念记录；
- 未来的服装、乐器和空间展示。

伙伴成长不能影响分析准确率、课程权限或核心教学公平。

### 8.3 首版成长不使用单一经验值刷取

首版建议使用少量可解释成长节点，例如：

- 完成第一次“发现问题 → 范唱 → 重唱进步”闭环；
- 完成第一次音程模唱；
- 完成第一次完整节奏练习；
- 让伙伴演唱用户创建的第一段旋律；
- 连续完成若干不同技能，而不是机械重复同一道题。

不因在线时长、连续签到、广告观看或付费直接提高伙伴教学能力。

## 9. 内容与台词架构

### 9.1 台词不能直接写死为一套

建议内容结构至少包含：

- `event`：发生了什么；
- `evidence`：底层事实；
- `teaching_intent`：要解释、示范、鼓励还是建议休息；
- `base_personality`：伙伴核心人格约束；
- `learning_style`：当前风格调节；
- `relationship_stage`：初识、熟悉或长期陪伴；
- `safety_constraints`：禁用羞辱、依赖诱导、虚假判断等；
- `locale`：语言与术语版本。

### 9.2 同一事实的表达示例

底层事实：第三小节最后一个音偏低，进入晚。

不同学习风格只改变表达方式：

- 温柔讲解：更完整、更安抚；
- 活泼互动：快速提出两音接唱；
- 精简专业：直接给出音高和节奏位置；
- 安静陪伴：只显示关键一句并等待用户主动重试。

首发伙伴的人格确定后，再在以上四种表达中加入其固定措辞、动作和情绪范围。

## 10. 数据与接口预留

未来实现时建议至少预留以下概念，不要求 M0 单独做空 schema PR：

- `CompanionDefinition`：伙伴身份、人格边界、资源和版本；
- `PersonalityProfile`：核心特质、上下限、标志行为和禁用行为；
- `LearningStyleProfile`：教学风格调节参数；
- `CompanionRelationship`：伙伴独立成长与纪念记录；
- `CompanionContext`：当前页面、练习、证据、权限和能力状态；
- `CompanionEvent`：开始、提交、错误、进步、中断、低置信等事件；
- `CompanionAction`：播放、定位、循环、调整速度和启动复练；
- `CompanionUtterance`：事实、教学意图、人格与风格组合后的输出；
- `CompanionSettings`：显示模式、风格、动画、语音和主动程度。

所有对象必须版本化，并允许未来增加新伙伴而不迁移或覆盖用户的核心音乐学习数据。

## 11. 首发测试指标

首发伙伴不能只用“用户觉得可爱”判断成功。

至少观察：

- 首次选择学习风格的完成率；
- 用户能否理解“人格不变、风格可调”；
- 练习完成率与复练率变化；
- 用户从错误反馈进入定向复练的比例；
- 用户主动隐藏、静音或降低互动的比例；
- 四种学习风格的选择和切换情况；
- 同一伙伴切换风格后的角色一致性感知；
- 动画和语音对音频延迟、耗电、温度和崩溃的影响；
- 用户是否将伙伴反馈误认为未经验证的专业诊断；
- 新鲜感消退后的持续使用情况。

### 11.1 定性研究问题

测试后需要询问：

- 你觉得四种模式下还是同一位伙伴吗；
- 哪些表现让你觉得角色前后矛盾；
- 哪种风格更愿意长期使用；
- 什么时候伙伴显得打扰；
- 什么时候伙伴确实帮助你理解了错误；
- 成长是否让你感到共同进步，而不是被迫签到；
- 你是否期待选择其他性格的伙伴，期待的差异是什么。

## 12. 未来扩展多伙伴的触发条件

只有在首发伙伴至少证明以下结果后，才进入第二位伙伴的正式制作：

- 用户能理解并使用学习风格；
- 四种风格没有破坏角色一致性；
- 伙伴能真实提高至少一项核心学习任务指标；
- 角色不会干扰专业音频和核心练习；
- 成长机制不依赖惩罚、付费或无意义刷取；
- 内容生产、动画、语音和测试成本已经可估算；
- 数据结构可以增加新伙伴而不重构核心项目与学习画像。

第二位伙伴应选择与首发伙伴明显不同的核心人格和轮廓，用于验证同一学习风格在不同先天人格下的差异，而不是只增加相似外观。

## 13. 与 M 泳道的对应关系

### M0：现在记录，恢复开发后做最小预留

- 使用单伙伴结构，但数据模型支持未来多伙伴；
- 人格层与学习风格层分离；
- 首发角色形象、故事和台词保持 `TBD`；
- 在 P113/P114/F1/F2 中预留事件、证据与工具边界。

### M1：首发测试伙伴

- 一只伙伴；
- 四种学习风格；
- 基础状态、唱名、短范唱和具体纠错；
- 完成第一条可信陪练闭环；
- 开始验证伙伴独立成长。

### M2：深化首发伙伴

- 接唱、模唱、找错、节奏互动；
- 冻结首发伙伴完整人格、视觉、故事和台词体系；
- 根据真实测试结果调整学习风格参数。

### M3 以后：验证后扩展

- 在工具型智能体稳定后评估第二位伙伴；
- 多伙伴共享音乐大脑、工具协议和安全边界；
- 每位伙伴保留独立人格、动作、声音、关系和成长；
- 用户可拥有多个伙伴并选择当前主陪伴伙伴；
- 切换伙伴不清空用户音乐能力，伙伴关系分别保存。

## 14. 当前结论

项目已确定采用：
