# 自动工具深度参与项目设计协议

状态：**Canonical governance protocol / 人工负责、工具深度参与**  
生效日期：2026-07-23  
适用范围：产品需求、路线、架构、交互、验收、实现、测试、证据准备与 P119/Q 抽样设计

## 1. 决策

项目允许自动工具高度参与项目设计与开发。自动工具不再被限制为只验证人工已经选定的方案，可以主动：

- 分析当前仓库事实、用户目标、依赖、风险和竞品能力；
- 提出产品范围、优先级、架构、交互、数据协议与验收标准候选；
- 比较多个方案并给出有依据的推荐；
- 在授权范围内生成实现、测试、迁移、文档和可回滚交付物；
- 为证据采集设计分层、样本量、可复现抽样方法和候选清单；
- 发现阻塞时提出替代路线，而不是无期限停在规划状态。

工具输出属于可审查的设计或实现候选。允许深度参与不等于工具可以把模拟结果、自动测试或自身判断冒充外部事实。

## 2. 默认工作模式

采用以下闭环：

```text
读取真实状态
→ 工具提出设计与依据
→ 自动检查约束、风险和可回滚性
→ 生成候选实现或候选证据计划
→ 人工在规定闸门明确批准
→ 执行、验证、记录 provenance
→ 依据真实结果继续迭代
```

普通、可逆、仓库内实现细节可以由工具自主决策。遇到产品范围、正式质量阈值、隐私、安全、破坏性数据、公开发布、外部人员结论或最终发布批准时，必须保留对应人工闸门。

## 3. 工具可主导的事项

在不越过第 4 节闸门的前提下，工具可以主导：

1. 需求拆解、路线排序、依赖分析和完整 vertical slice 设计；
2. 技术架构、schema、迁移、错误恢复、测试与回滚方案；
3. UI 信息架构、简体中文文案候选、空/错/加载/禁用状态；
4. 自动化测试、静态检查、构建、CI、证据目录和状态同步；
5. 可复现的分层抽样候选、随机种子方案、算法版本和候选 item ID；
6. 风险登记、未完成项识别、诚实状态回报与下一步建议。

工具应优先选择确定性、可复现、可审计的方案；使用随机过程时必须记录输入清单摘要、分层规则、样本量、随机种子、算法名称/版本和输出摘要。

## 4. 必须保留的人工责任

下列事项不能因工具高度参与而自动视为完成：

- 产品所有者对新增 V1 MUST、降低质量门槛、重大范围变更和最终发布的批准；
- 公开披露敏感信息、生产权限、Secret、不可逆数据操作和破坏性迁移的授权；
- 教师、研究参与者、目标用户、许可方或其他外部人员的身份、资质、同意、签署和真实结论；
- 真实 Android、真实麦克风、真实人声、真实竞品同机任务和真实用户可用性证据；
- 正式教育有效性、评分、公平性、法律、医疗或安全结论；
- 将自动草稿提升为正式目标、正式成绩或公开发布结论。

工具可以准备材料、检查结构、汇总匿名结果和指出缺口，但不得伪造上述证据或代表外部人员签署。

## 5. P119/Q 抽样专用规则

P119/Q 允许自动工具深度参与抽样设计和候选名单生成：

1. 工具可以根据冻结 manifest 提出样本量与分层方案；
2. 产品所有者先批准样本量、分层规则和抽样方法；
3. 工具可以使用已批准的规则、公开记录的随机种子和版本化算法生成候选 item ID；
4. 输出必须绑定 source commit、catalog version、manifest 文件和 SHA-256，并记录种子、算法、每层数量、候选 ID 列表及列表摘要；
5. 产品所有者必须在教师开始审核前查看并明确批准最终精确 item ID；
6. 两名独立教师审核同一冻结批次；工具不能填写教师判断、签署或开放问题结论；
7. 任何输入、算法、种子或清单变化都使旧候选批准失效，必须重新生成和批准。

因此，自动工具可以设计并执行可复现选样，但不能批准自己的候选名单，也不能把候选生成写成教师审核已经执行。

## 6. provenance 与诚实状态

每个由工具主导的重要决定至少记录：

- 使用的仓库基线、输入文件和版本；
- 目标、假设、约束与被排除方案；
- 选择理由和已知风险；
- 自动生成算法、参数、随机种子和输出摘要（如适用）；
- 哪个节点需要哪位责任方批准；
- 已执行、未执行和被阻塞的验证；
- 回滚或重新生成方式。

自动测试、CI、Debug APK、模拟音频、自动清单与工具评审继续分别记录，不能替代真实设备、真实人员或教育证据。

## 7. 与现有规范的关系

本协议扩大工具参与设计和候选生成的权限，不降低以下规范：

- `docs/final-release-definition-of-done.md` 的正式发布门槛；
- 自动结果“预览 → 检查 → 修改／确认 → 再进入练习”的原则；
- 隐私、所有权、删除、许可、低置信拒答和非评分边界；
- GitHub PR、CI、真实 QA 分层、可回滚与证据可追溯要求；
- 社区、公开上传、公开曲库、支付等现有范围限制。

如其他路线仍写着“工具不得参与设计或生成候选”，以本协议为新的治理基线；但具体领域的人工最终批准和真实外部证据要求继续有效。
