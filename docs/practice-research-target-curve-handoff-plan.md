# P16h Practice Research Target Curve Handoff Plan

## Scope

P16h is a docs-only handoff planning step. It documents how a future runtime PR may safely and explicitly move a `ResearchTargetPitchCurveDiagnostic` generated by `/research/local-audio-decode` into a read-only preview on `/practice`.

P16h does **not** implement the handoff. It does not change `/practice`, `/research/local-audio-decode`, storage, scoring, audio import, or Practice Mode target behavior.

## 1. Source Decision

Selected conservative approach:

**Option A — explicit manual JSON copy/paste handoff for research target curve diagnostics.**

Reasons:

1. `/research/local-audio-decode` already generates a `ResearchTargetPitchCurveDiagnostic`.
2. `/practice` already displays a same-family diagnostic preview shape.
3. The safest next connection should be an explicit manual copy/paste handoff, not automatic handoff.
4. Manual JSON handoff does not require accounts, databases, server storage, uploads, cloud services, or AI APIs.
5. Manual JSON handoff can preserve the local/research-only boundary.
6. `/practice` can parse and display the pasted diagnostic as read-only preview without replacing mock melody targets.
7. This is closer to real usage than a static fixture while still not pretending to be formal Practice Mode audio import.

## 4. Input Contract

A future `/practice` paste panel may accept only research-only JSON matching the P15b output shape.

`ResearchTargetPitchCurveDiagnostic`-like input:

- `curveId`
- `source: "local-audio-decode-note-like-segments"`
- `segments[]`
  - `segmentIndex`
  - `startTimeSeconds`
  - `endTimeSeconds`
  - `durationSeconds`
  - `targetFrequencyHz`
  - `targetNoteLabel?`
  - `diagnosticConfidence: "normal" | "low"`
  - `sourceFrameCount`
  - `bridgedNullFrameCount`
- `summary`
  - `segmentCount`
  - `totalDurationSeconds`
  - `lowConfidenceSegmentCount`

## 5. Validation / Guardrails for Future P16i

If a future P16i implements paste preview, validation must remain conservative:

1. JSON parse failure displays a non-scoring error.
2. Missing required fields displays invalid diagnostic JSON.
3. `diagnosticConfidence` allows only `normal` or `low`.
4. `high` / `medium` must be rejected, not normalized, to avoid reintroducing grade-like or level-like confidence implications.
5. `source` must exactly match the expected research source.
6. `segments` must be an array.
7. Segment numeric fields must be finite.
8. `targetFrequencyHz` must be a positive finite number.
9. `summary` may be recalculated or cross-checked.
10. Invalid input must not affect the existing mock melody flow.
11. Invalid input must not write attempt history.
12. Invalid input must not trigger pitch comparison.
13. Invalid input must not trigger scoring.
14. Invalid input must not trigger network behavior.

## 9. P16i Implementation Note — Isolated Manual JSON Handoff Parser

P16i implements only the isolated runtime helper for a future manual paste handoff. It does not add a copy button, paste panel, `/practice` UI integration, `/research/local-audio-decode` UI integration, storage, scoring, target replacement, or real audio import.

The helper accepts only JSON text as a `string` and returns an explicit result union:

- `ok: true` with a `ResearchTargetPitchCurveDiagnostic` when the pasted text matches the conservative research-only contract.
- `ok: false` with an error code and non-scoring diagnostic message for empty input, invalid JSON, invalid shape, unsupported source, unsupported confidence, or invalid numbers.

P16i uses strict summary validation instead of implicit summary normalization. `summary.segmentCount` must equal `segments.length`, and `summary.lowConfidenceSegmentCount` must equal the number of `low` diagnostic segments. This avoids silently rewriting pasted diagnostics.

The helper validates only the research source `local-audio-decode-note-like-segments`; segment numeric fields must be finite; `targetFrequencyHz` must be positive; frame counts must be non-negative; `diagnosticConfidence` must be exactly `normal` or `low`; `high` and `medium` are rejected; optional `targetNoteLabel` must be a string when present; and `summary.totalDurationSeconds` must be finite and non-negative.

Synthetic tests cover empty input, invalid JSON, missing top-level fields, unsupported source, valid empty / one-segment / multi-segment diagnostics, normal and low confidence acceptance, high and medium confidence rejection, non-finite numbers, negative target frequency, missing segment fields, optional note label omission and string acceptance, summary count mismatches, returned diagnostic copy isolation from parsed input, and non-scoring error wording.

## 10. P16j Runtime Beta Slice — Manual JSON Copy/Paste Preview

P16j implements the minimal research-only manual JSON handoff beta UI from `/research/local-audio-decode` to `/practice`:

1. `/research/local-audio-decode` shows a **Copy diagnostic JSON** action only after a `ResearchTargetPitchCurveDiagnostic` exists from the existing explicit Extract pitch frames flow.
2. The research route exposes the same diagnostic JSON in a read-only textarea fallback for manual copying.
3. `/practice` adds a **Manual research target curve diagnostic JSON preview** panel that accepts pasted JSON and previews it read-only after explicit user action.
4. The `/practice` paste panel uses the P16i `parseResearchTargetCurveHandoffJson` helper without changing parser semantics.

P16j remains research-only. It is not automatic import, not formal Practice Mode audio import, not formal TargetPitchCurve integration, not target replacement, not scoring, not assessment, not melody recognition, not note recognition, not Song Learning Mode, and not APK-ready.

The runtime slice intentionally does not use `localStorage`, `sessionStorage`, URL payload handoff, APIs, upload, cloud, AI calls, account data, database data, or server storage. The imported preview is read-only and does not affect mock melody targets, `currentMelodyStepIndex`, target note playback, pitch estimation, target-aware comparison, or attempt history.
