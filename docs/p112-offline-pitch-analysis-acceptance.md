# P112 录音提交后本地多候选音准分析验收边界

更新：2026-07-18

QA level recommendation：**strict**

## 已交付切片

P112 在现有显式会话录音之上增加独立的录音后分析，不复用实时曲线冒充后处理结果：

1. 用户停止录音后先检查说明并二次确认；确认前不得解码。确认时停止实时麦克风和回放。
2. 本机 `AudioContext.decodeAudioData` 解码后，下混 1–8 声道、移除 DC、只在峰值超过 1 时衰减，并确定性线性重采样为 16 kHz 单声道 PCM。单次上限 30 秒。
3. 每个 2048 样本帧以 320 样本步长同时运行仓库自相关基线与 `pitchy@4.1.0` McLeod 候选，记录各自频率和置信度。
4. RMS 门控、候选阈值、引擎分歧、超范围和无法解释的大跳均可产生显式拒答；不强迫每帧输出音名。
5. 两候选先按八度等价关系比较，再做置信加权的对数频率合并；连续轨迹只在整八度候选可回到前帧 ±180 cents 时修正。算法不读取目标音，因此不会把唱错强行吸附成正确。
6. UI 显示代表音高、可靠覆盖、候选一致、拒答、八度修正、削波、处理毫秒、RTF 和连续轨迹。结果只在当前页面会话；丢弃或新录音立即失效。

## 算法版本和确定性

- 版本：`offline-pitch-multicandidate-v1`
- 标准采样率：16,000 Hz
- 帧 / 步长：2048 / 320 samples
- 观察范围：65–1100 Hz
- 同一标准 PCM 和同一代码版本必须产生逐帧确定性等价结果；实际墙钟处理毫秒和 RTF 是设备诊断，不属于确定性核心结果。

## 明确不做

- P112 不实现 P113 的音符分段、目标节拍/逐音对齐、音头/稳定段/尾音反馈或分段复练。
- 不把代表音名、可靠覆盖或候选一致率写成正式成绩、等级、通过/失败、声部判断或教学结论。
- 不上传录音或 PCM，不调用云端、AI、识别 API，不自动把普通练习录音加入 P111 数据集。
- 当前两个候选和合并阈值尚未通过 P104/P111 真实人声盲测；“多候选”不等于已证明专业准确。

## 自动退出门槛

- 48 kHz 双声道带 DC 的 A4 可标准化并由两个候选稳定识别；静音全部拒答。
- 八度候选连续修正、超长/过短/空声道拒绝、相同 PCM 确定性等价均有纯逻辑测试。
- 真实挂载测试覆盖“录音 → 停止 → 检查 → 二次确认 → 本地解码 → 结果 → 丢弃失效”，确认前不得调用解码。
- `lint`、`typecheck`、完整运行测试、移动构建、Android 本地 bundle 校验和 Next.js build 通过。

## 真机与外部证据

当前开发退出允许使用 CI debug APK 与自动测试，但三档 Android 性能仍保持 `IN_PROGRESS`。按产品所有者决定，低/中/高完整矩阵在 P118 功能完整评估 APK 后执行。每台需记录录音时长、解码+分析毫秒、RTF、Android、System WebView、采样率、输入场景和 APK SHA-256；目标是 RTF P95 不高于 0.5。缺少三档真实记录时不得宣称性能门槛 PASS。
