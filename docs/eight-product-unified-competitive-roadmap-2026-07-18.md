# 八产品全能音乐学习系统竞争路线：练声、音准、视唱练耳、乐理、钢琴、伴奏与制谱

决策分析日期：2026-07-18
对象：用户提供的八款目标 APK、当前项目事实与公开官方资料
性质：**Canonical product rebaseline / 八产品长期竞争路线**；是产品与验收路线，不是功能完成证明
当前执行边界：P106–P112 已合并；项目暂停在 P113 开始前，本报告不代表已启动任何新运行时切片

## 1. 本次范围纠正与最终定位

### 1.1 正式对标对象为八款

1. 《开嗓练声》；
2. 《知唱音域音调仪》；
3. EarMaster；
4. 《来音乐理》；
5. 《来音钢琴》；
6. 《来音伴奏》；
7. 《来音制谱》；
8. 《完美钢琴》（Perfect Piano）。

《练耳大师》（`com.ggyd.EarPro`）明确移出目标清单。此前对它完成的静态分析只保留为旧式 FFT 音高算法和早期活动界面的历史基线，不再从它继承产品需求，也不把它计入八款。

### 1.2 最终产品不是“八个入口”，而是一套完整学习系统

产品目标应正式表述为：

> 面向中文用户的全能音乐学习与创作系统。用户可从一首歌、一份乐谱、一个课程知识点或一次自由演奏开始，在同一个项目中完成制谱、识谱、听辨、视唱、练声、钢琴演奏、伴奏处理、录音分析和定向复练。每个模块单独达到优秀单项产品的核心体验，同时由统一乐谱、音频、练习目标、历史和学习画像形成协同优势。

这意味着两个门槛必须同时成立：

1. **单项替代门槛**：只使用某一模块时，目标用户没有明显理由立即切回对应竞品；
2. **整体协同门槛**：该模块产生的乐谱、音频、分析、错误和进度能被其他模块直接复用。

“全能”不能成为单项质量一般的借口。钢琴延迟、音色、触控或 MIDI 不好，用户不会因为还有练耳而留下；制谱丢符号、导出失真，用户不会因为还有音调仪而迁移；音准曲线误报且不拒答，也会破坏整个系统的可信度。

### 1.3 本地与云端都要优秀

离线不是最终卖点，只是当前中国区服务、合规、基础设施和运营条件尚未准备完成时的开发边界。

- 本地负责低延迟演奏、实时音高、基础课程与题目、录音、基础识谱/制谱、隐私敏感处理、断网连续性和可预测降级；
- 云端负责账号与多端同步、课程/题库/谱库分发、教师与作业、复杂 OMR、重型音频分析、多轨分离、备份恢复、模型/音色更新和经过授权的数据分析；
- 同一能力可以同时有本地、云端和混合 provider，由设备、网络、地区、用户同意、质量与成本决定；
- 云端不可用时已下载核心功能应继续工作，但这不等于最终产品只是离线工具；
- 当前代码不得把特定境外服务、域名或厂商写死进钢琴、制谱、练习和项目数据模型。

## 2. 分析方式、合法边界与结论强度

本次对新上传的《完美钢琴》和《来音制谱》执行了只读静态分析：核对 Manifest、资源、DEX/smali、Flutter AOT 字符串、ELF 依赖与符号、模型/音色文件、导入导出入口和网络路径。两份 APK 都通过 ZIP 全量完整性检查。

没有安装或运行未知 APK，没有登录账号，没有绕过加密、付费、反调试或服务器限制，也没有提取、复制或复用竞品代码、模型、音色、曲库、课程、题目、图标或 UI。

证据分为三类：

- **确认**：APK 中的调用、符号、资源、常量或官方公开资料可以直接证明；
- **高置信推断**：多个独立证据一致，但缺少运行时参数或服务端实现；
- **未知**：必须进行合法真机黑盒测试、取得厂商资料或使用标注数据才能判断。

静态分析不能证明准确率、延迟、音质、卡顿率、崩溃率、课程正确率或用户满意度。任何“比竞品更准、更快、更专业”的公开表达，必须来自同设备、同输入、同真值、冻结版本和可重复报告。

## 4. 完美钢琴：成熟移动演奏器的强项

### 4.1 音频与音色实现

APK 确认包含：

- `libbass.so` 与 `libbassmidi.so`，以 BASS/BASSMIDI 驱动 SoundFont 合成；
- 六个内置 SF2，资源可辨识为三角钢琴/另一钢琴、音乐盒、管风琴、Rhodes 电钢琴和合成贝斯等类别；
- MIDI stream、SINC 插值、混响与音量等本地合成路径；
- 内部配置尝试把 MIDI voices 设为 1000，失败时回退 500。它只能证明配置上限，不能证明真实手机能无卡音达到这一复音数；
- 低延迟音频开关以及 OpenSL ES/BASS 设备缓冲相关配置，但真实手势到扬声器延迟仍须真机测量。

这条路径的价值在于成熟、轻量和可扩展，而不是证明它的默认钢琴音色一定优于来音钢琴。其采样资产和许可不能复制；我方必须使用独立合法音色。

### 4.2 演奏与键盘体验

官方页面与 APK 共同确认：

- 88 键逻辑键盘；
- 单排、双排、双人和和弦模式；
- 多点触控与可用时的压力触控；
- 键宽调整与键盘总览快速定位；
- 音名标签可选科学音高、唱名/数字、仅 C 或关闭；
- 延音长度、混响、主音量与立即控制；
- 单/双排及横向演奏的成熟布局。

完美钢琴在这里给出的不是“有一排键”，而是移动触屏钢琴经过长期打磨后的可操作性基线。

### 4.3 录音、MIDI 与学习

确认或官方公开列出的能力包括：

- MIDI 事件、麦克风音频及组合录制入口；
- 录音管理、重命名、删除、导出/分享；
- 节拍器 50–250 BPM，至少包含 3/4、4/4、6/8；
- USB MIDI，外部键盘可用于演奏、录制和学习；
- 免费音色插件与自定义 SF2 方向；
- MIDI、MusicXML/MXL/XML 导入，`libxml2midi.so` 明确把 MusicXML 转为 MIDI，并处理音符、速度、拍号和踏板事件；
- 落键、瀑布、五线谱三种指导视图；
- 自动播放、半自动/可见音练习、错音等待；
- 左右手、A–B 循环、速度和难度调整。

### 4.4 不应自动纳入的外围

APK 还包含公开社区、好友、聊天、公会、排行榜、周挑战、多人比赛、公开 MIDI/音频/视频上传、金币、广告和支付等大规模商业/社交能力。

这些与当前 canonical 路线中“暂不做社区、聊天室、公会、排行榜、多人竞技和公开曲库”直接冲突。本报告不会把它们偷偷塞进钢琴工程量，也不会假装用户本次“功能齐全”已经自动解决该冲突。默认仍把**演奏、学习、MIDI、录音和资源扩展**视为完美钢琴的核心对标范围；若产品所有者要撤销社交/公开内容排除，必须另立内容治理、版权、未成年人、审核、反滥用和运营路线。

## 5. 来音钢琴与完美钢琴：取长后的统一钢琴目标

### 5.1 两者的优势并不相同

| 能力 | 完美钢琴更强的证据 | 来音钢琴更强的证据 | 我方组合目标 |

| 触屏键盘 | 单/双排、双人、和弦、键宽、标签、压力和键盘总览成熟 | 有虚拟键盘、横竖屏、粒子/掉音等表现 | 完整保留完美钢琴式键盘广度，并做原创交互与无障碍 |

| 学习内容 | 落键/瀑布/五线谱、三种练习模式、A–B/分手/速度 | 谱库、课程、掉音游戏、AI 陪练结果、练习单元更完整 | 谱面、瀑布、掉音至少三种原创视图；课程、分手、循环、逐音证据统一 |

| 谱面入口 | 本地 MIDI/MusicXML 导入与学习 | MusicXML、PDF/图片/OMR、谱库和课程联动更强 | 所有导入先进入乐谱工作台检查确认，再进入钢琴学习 |
| 练习反馈 | 有完成/准确率等结果，但透明度未知 | 音高、节奏、速度、错误统计更丰富 | 不用一个总分掩盖错误；定位到音符、小节、手别、时间与置信度 |
| 录制与管理 | 本地录音/文件管理和导出成熟 | MIDI/屏幕录制、练习回放 | 事件序列＋音频＋项目自动保存；可编辑、重命名、导出与恢复 |
| 扩展 | SF2/音色插件清晰 | 与来音谱库、课程、OMR、其他乐器 App 联动 | 合法音色/课程/模型资源包＋统一乐谱和学习系统 |

### 5.2 我方钢琴最终功能范围

1. A0–C8 全 88 键，连续滚动、键盘总览、快速八度定位；
2. 单排、双排、双人、和弦快捷和练习内嵌紧凑模式；
3. 至少 10 指状态正确；生产至少 32 音复音，目标 64；
4. 键宽、标签、中央 C、移调、横竖屏、左右手颜色与无障碍；
5. 可用时把触控压力/接触面积映射为 velocity，无能力时使用稳定默认力度；
6. 默认合法采样三角钢琴，至少 3 力度层、自然衰减和无爆音释放；
7. 至少六类内置可辨识音色，并支持版本化可下载音色/SF2 包；
8. 延音、踏板、半踏板能力检测、基础混响、主音量、全停和生命周期清理；
9. 节拍器、预备拍、事件录制、音频录制、回放、重命名、删除与导出；
10. A–B、速度、左右手/声部过滤、谱面、瀑布和掉音视图；
11. USB MIDI、BLE MIDI、软件 MIDI、velocity、channel、sustain pedal 和断连恢复；
12. MusicXML/MXL/MIDI/确认后的 OMR 乐谱进入学习；
13. 课程、谱库与自由项目都能进入同一练习器；
14. 每次练习可回到具体错音、漏音、多弹、时值/节拍偏移和对应原始事件。

### 5.3 钢琴硬门槛

- 调律误差不超过 ±2 cents，A4 支持 415–466 Hz；
- 手势到音频计划启动 P95 ≤80 ms，扬声器、耳机和蓝牙实际输出延迟分别报告；
- 首次可弹 P95 ≤2 秒；
- 10 指、32 音＋延音压力测试无可复现卡音、爆音、常驻声源或无界音量叠加；
- 三档 Android 记录 CPU、内存、热降频、电量和音频 underrun；
- 低端降级只能降低复音、混响或力度层，必须明确提示，不能静默退回廉价单振荡器；
- MIDI 断连、切后台、来电、音频焦点丢失、旋转与组件卸载后无残留声；
- 音色盲听与演奏可用性必须在目标用户和教师样本上验证，不能只靠资源大小宣称音质。

## 6. 来音制谱：完整制谱工作台与算法边界

### 6.1 APK 已确认的编辑能力

来音制谱不是一个孤立 OCR 页面。APK 中可确认完整的 score/part/staff/voice/measure/note 数据结构、手工制谱页面、撤销/重做、复制/粘贴、移动、选择和播放控制。

内置 18 类模板：低音五线谱、四线贝斯谱、鼓谱、吉他、吉他＋声乐、古筝、古筝＋钢琴、古筝单谱、钢琴、钢琴＋简谱、双钢琴、钢琴＋小提琴、钢琴＋声乐、尤克里里、尤克里里＋声乐、小提琴、声乐和双声乐。

确认存在以下记谱域：

- 五线谱、简谱、吉他/贝斯六线/四线谱、鼓谱、古筝专用符号和弹唱谱；
- 多 part、多谱表、多声部、小节、音符/休止符、附点、连音、符杠与多小节休止；
- 高音/低音/中音/次中音/打击乐/tab 谱号，调号、拍号、散板/弱起、速度与速度变化；
- 升降还原、连音线、延音线、倚音、震音、琶音、波音、颤音、回音、滑音和踏板；
- 强弱、渐强渐弱、重音、断奏、保持、呼吸、弓法、拨弦、弱音器和演奏文字；
- 反复、房子、布局换行、乐段/组括号；
- 歌词、指法、和弦名称/图表、转位/省略/变化音；
- 吉他 capo、击弦等技法；古筝按弦、滑音、摇指等专用输入；
- 全谱/分谱、整体移调、半音/音程移调和五线谱/简谱展示转换。

### 6.2 多种输入、播放和导出

APK 证实：

- 屏幕符号面板、数字键盘、虚拟钢琴键盘；
- USB/BLE/虚拟 MIDI；
- 实时 MIDI 转谱，包含量化、节奏模式匹配、声部估计、节奏修复和布局构建类；
- 本地 TFLite 音频到音符事件链路，模型资源约 13.96 MB；
- FluidSynth/SoundFont 乐谱播放、和弦播放、节拍器、混响/合唱和 sustain 状态；
- `lysc`、MusicXML/XML/MXL、MSCZ、OVE、MIDI 等导入/转换入口线索；
- 图片、MIDI、MP3、PDF、同步视频和 XML/MusicXML 等导出图标/代码入口；
- 全谱与分谱、保存副本、文件夹、复制、移动、上传、分享和同步冲突模型。

专有或第三方格式只能通过合法公开规范、授权 SDK 或用户拥有的文件实现，不得从竞品复制解析器。正式支持列表应按我们的 round-trip 证据逐项开放，而不是因 APK 中出现字符串就全部宣称完成。

### 6.3 OMR 的真实本地/云端边界

本地端确认存在：

- `libstaffdetect.so`；
- 约 6.1 MB 的加密 `staff_detect_20260304_enc` 模型；
- ONNX Runtime；
- `StaffDetector::InitOrt/Detect`、相机预览、拍照、bitmap/buffer 检测和谱表检测通道。

同时，Flutter AOT 明确包含：

- `/music/score/omr/oss`、`/music/score/omr/record`、`/music/score/query/omr/record`；
- `/music/score/omr/result/feedback`、`/music/score/omr/result/instrument`；
- `/smartscore/upload/editscore`、`/smartscore/upload/oss`；
- OMR 权益、消耗、失败日志、上传、任务记录和结果反馈等路径。

因此最可信的判断是：**本地模型负责谱表/页面区域检测、相机辅助或上传前预处理；完整拍照转可编辑乐谱至少存在一条云端上传—任务—结果链路。** 不能把这个样本宣传为“完整 OMR 全程本地”。也不能仅凭端点证明所有识别必然走云端；最终比例仍需动态测试。

### 6.4 我方应超过它的地方

1. OMR 不是孤立 OCR，而是“导入—置信度—修改—确认—练习”的第一步；
2. 本地快速 OMR 与云端高质量 OMR 使用同一结果 schema，用户明确选择处理位置；
3. 每个低置信符号、音符、时值、谱表/声部归属都有可定位证据，不把整页包装成一次成功/失败；
4. 保留原图/PDF、识别草稿、用户修改和确认版本，任何重新识别都不静默覆盖；
5. MusicXML/MXL 是开放交换主格式，内部项目另有版本化 schema；
6. 一份确认乐谱能直接进入钢琴、视唱、旋律听写、节奏练习、练声和伴奏；
7. 导出经过语义 round-trip 与独立阅读器验证，而不是“能生成一个文件”即通过；
8. 云端同步有离线队列、冲突比较、版本历史、恢复、导出与彻底删除；
9. 对简谱、中文歌词和古筝等中文用户高频场景建立独立数据集与验收，不只优化西方单谱表钢琴谱。

## 7. 我方“乐谱工作台”正式目标

### 7.1 创建与导入

- 从空白或多乐器/编制模板创建；
- 导入 MusicXML/MXL、MIDI、PDF、图片、相机扫描；
- 在合法可行时支持 MSCZ、OVE 等格式转换；
- USB/BLE MIDI 实时记谱；
- 麦克风/音频产生音符候选；
- 从伴奏模块的 BPM、拍点、调性、和弦与旋律候选建立草稿；
- 所有自动结果必须先预览、检查、修改/确认，再成为正式练习目标。

### 7.2 编辑与排版

- 完整 part/staff/voice/measure/note 模型；
- 五线谱、简谱、tab、鼓谱、古筝谱和弹唱谱；
- 常用音符、时值、调拍谱号、演奏法、力度、装饰、反复、歌词、和弦、指法与专用乐器符号；
- 多选、复制/剪切/粘贴、拖动、批量移调、声部/分谱管理；
- 撤销/重做、自动保存、崩溃恢复、版本比较；
- 页面尺寸、边距、系统间距、换行/分页、标题/作者/版权和打印预览；
- 原创移动端高效输入，不复制来音制谱的键盘、图标或页面布局。

### 7.3 播放与练习

- 合法采样音色、逐声部 mute/solo/音量/声像；
- 节拍器、预备拍、速度、移调、循环、定位和踏板；
- 事件时间轴与谱面光标严格同步；
- 一键进入钢琴谱面/瀑布、视唱、节奏、听写、练声或伴奏；
- 用户练习结果可反向标注到小节和音符，但不得破坏原谱。

### 7.4 导出与协作

- 内部项目包、MusicXML/MXL、MIDI；
- PDF、PNG/JPEG/SVG 等视觉导出；
- WAV/MP3 等音频导出；
- 同步谱面视频；
- 全谱/分谱分别导出；
- 本机项目在当前阶段完整可用；服务条件成熟后增加同步、版本历史、教师批注和受控协作；
- 公开谱库、公开上传和陌生人协作仍受当前范围排除，除非产品所有者另行授权。

### 7.5 制谱与 OMR 建议质量门槛

以下是我方拟定门槛，不是宣称竞品已经达到：

- 对正式支持的 MusicXML 子集，音高、时值、小节、调号、拍号、谱号、声部和歌词语义 round-trip 100% 等价；不支持元素必须显式列出；
- 建立不少于 500 份合法/自建乐谱 fixture，覆盖单谱表、钢琴大谱表、多声部、多 part、简谱、tab、鼓谱、歌词、和弦、反复与复杂符号；
- 常用编辑动作在中档 Android P95 ≤100 ms，后台自动保存不阻塞演奏/编辑；
- 连续撤销/重做、崩溃恢复和同步冲突不丢已确认版本；
- PDF/图片 OMR 分别报告音符音高 F1、时值 F1、小节结构、谱表/声部归属、歌词识别、符号检测和拒答率；
- 清洁印刷谱的目标门槛可先定为音高 F1 ≥0.98、时值 F1 ≥0.95；复杂多谱表、斜拍、阴影、低清、简谱和手写样本单独分层，不用总体均值掩盖失败；
- OMR 还要报告“修正到可练状态的时间”，目标是在支持场景中较纯手工录入中位数至少减少 50%；
- 任何页面在置信度不足、裁切不全、透视严重或格式不支持时局部/整页拒答，不能生成高置信错误谱；
- 使用 MuseScore 等独立合法阅读器做交叉打开测试，但不以第三方渲染像素完全一致作为唯一正确性标准。

## 9. 真正的竞争壁垒：四条跨模块黄金闭环

### 9.1 乐谱到完整学习

```text
PDF / 图片 / MusicXML / MIDI / 手工输入
→ 本地或云端识别/转换草稿
→ 每符号置信度与原图对照
→ 用户修改并确认版本
→ 钢琴分手/降速/循环
→ 旋律视唱、节奏视读、听写与乐理解释
→ 录音或 MIDI 结果对齐到音符/小节
→ 错误直接进入定向复练
```

这条闭环把项目最初的识谱方向升级为真正有持续价值的学习入口。

### 9.2 歌曲到练会

```text
导入个人音频/视频
→ 本地快速或云端高质量分离
→ BPM / 拍点 / 调性 / 和弦 / 段落候选
→ 根据音域建议移调
→ 用户预览确认
→ 分句跟唱或跟弹
→ 停止后逐音逐句重分析
→ 一键循环薄弱句
→ 可生成/修正乐谱草稿
```

### 9.3 课程知识到可演奏能力

```text
中文知识点
→ 听辨
→ 五线谱/简谱/钢琴/唱名作答
→ 自己唱或弹
→ 解释具体错误
→ 自动生成一小节谱例
→ 加入技能画像与复习计划
```

### 9.4 自由创作到学习资产

```text
钢琴/MIDI/哼唱录入
→ 量化和制谱草稿
→ 用户编辑确认
→ 配和弦/伴奏/多音色播放
→ 导出或继续练习
→ 版本历史保留每次修改与演奏
```

单项竞品通常只覆盖其中一段；我方应让用户不再在多个 App 之间搬运文件、调性、速度、循环段、错误位置和练习记录。

## 10. 统一产品数据与技术架构

### 10.1 七个共享核心对象

1. `ScoreDocument`：part/staff/voice/measure/note、布局、歌词、和弦、来源、导入问题、确认修订；
2. `PracticeTarget`：乐谱、音频、调性、节拍、段落、循环、移调、速度和确认版本；
3. `MediaProject`：原始音视频、stems、片段、效果、录音、非破坏编辑图和导出；
4. `ActivityDefinition`：技能、难度、谱号/调号/拍号、答案模式、输入方式、解释和内容版本；
5. `AnalysisEvidence`：音高/节奏/演奏/OMR 结果、置信度、拒答原因、算法/模型版本和处理位置；
6. `LearningEvent/Profile`：课程、练习、错误、复练、建议原因、用户重置/关闭状态；
7. `ResourcePackage`：音色、模型、课程、题库、谱例的来源、许可、版本、SHA-256、大小、设备要求与删除入口。

### 10.2 统一输入事件

```text
触屏 / USB MIDI / BLE MIDI / 麦克风 / 音频文件
→ 标准 note-on/off、velocity、pedal、channel 与 PCM
→ 原生低延迟音频图
→ 音色 / 播放 / 录音 / 分离 / 分析
→ 乐谱、练习和证据层
```

钢琴、制谱、视唱、听写和伴奏不能各自再定义一套 MIDI、时间轴和音名规则。

### 10.3 本地/云端 provider

```text
能力请求
→ capability resolver（地区、网络、设备、登录、授权、质量、成本）
→ local provider | cloud provider | hybrid provider
→ 版本化结果＋来源＋置信度
→ 预览、比较、修改/确认
```

需要从现在就冻结的服务契约：Identity、Sync、Content、Media、Analysis、OMR、Recommendation、Telemetry。当前均可先由本机实现；未来接入中国区合规服务时不重写业务模型。

## 11. 安装包和资源策略：完整优先，不机械追求小

用户已经明确“安装包大一点没关系，功能一定要齐全”。据此调整原则：

- 不以首包体积为理由砍掉钢琴采样、基础课程、基础题库、核心 OMR/音高模型或必要导出能力；
- 默认安装至少包含可用的专业钢琴、基础多音色、音调仪、核心练习、手工制谱、基础本地识谱/分析和项目读写；
- 大型音色、完整课程、多语言内容、高质量 OMR、多轨分离和云端回退模型可做版本化资源包；用户可一次性预下载并永久断网使用已授权内容；
- “资源包”是更新、许可、设备适配和内存管理机制，不是把承诺过的功能变成空入口；
- 设置页显示每个包的功能、大小、版本、许可、下载/删除、校验和设备要求；
- 对低存储设备提供明确选择和失败恢复，不静默删除用户项目或降级音色；
- 仍需测安装成功率、首次启动、首次可弹、模型加载、资源迁移、增量更新和回滚，因为大包体会影响真实可用性。

- 知唱：持续音、音域、滑音、颤音、噪声、录音与曲线稳定性；
- EarMaster：音程/和弦/节奏/旋律/视唱活动、定制、统计和反馈闭环；
- 来音乐理：中文知识、题库、答案输入、解释、收藏和错题复练；
- 来音钢琴：谱库到练习、左右手、降速、循环、多输入和结果定位；
- 来音伴奏：导入、分离、变调/变速、多轨编辑、导出和进入练习；
- 来音制谱：创建、MIDI 输入、拍照识谱、修改、播放、导出和恢复；
- 完美钢琴：首次可弹、键盘模式、10 指、延音、录音、MIDI、插件和学习视图。

每项记录任务完成率、时间、步骤、错误恢复、延迟、资源、崩溃和主观负担。竞品分数不是算法真值；音高、OMR、分离和演奏必须对独立真值或原始事件。

### 13.2 模块与系统双门槛

| 模块 | 单项门槛 | 协同门槛 |
| --- | --- | --- |
| 音调仪/练声 | 快速、稳定、录音完整、准确度与拒答达标 | 一次录音可进入逐句分析、课程、歌曲与薄弱点复练 |
| 视唱练耳/乐理 | 14 活动族、中文课程、答案、解释、统计完整 | 知识点能被唱、弹、听写、制谱，错误进入统一画像 |
| 钢琴 | 音色、触控、延迟、MIDI、录制、学习视图达标 | 一份谱与错误小节可被视唱、听辨和伴奏复用 |
| 伴奏 | 分离、多轨、变调/变速、效果、导出可独立替代工具 | 处理项目可直接变为跟唱、跟弹、制谱与音域目标 |
| 制谱/识谱 | 编辑、排版、导入导出、OMR、播放、恢复可靠 | 确认乐谱无需重新录入即可进入全部练习模块 |
| 云端/课程 | 同步、内容、搜索、教师/作业和恢复可靠 | 本地与多端共享同一项目、版本、画像和证据 |

### 13.3 不得提前宣传的结论

- 未做同机标注数据前，不能说音高比知唱、开嗓练声或 EarMaster 更准；
- 未做外部回采前，不能说钢琴延迟或音质超过完美钢琴/来音钢琴；
- 未做 round-trip 和 OMR 数据集前，不能说制谱/识谱超过来音制谱；
- 未做合法 stems 和主观听测前，不能说分离超过来音伴奏；
- 未完成教师审核前，不能说课程、分数、等级或通过/失败具有专业教育效度；
- APK 内出现某个模型或库，不代表其准确率、训练质量或最终 UI 使用方式已知。

## 14. 用户获取与产品表达

不能诚实承诺“把八款软件的用户全部拿到”，但可以让每类用户都有迁移动机：

| 用户来源 | 第一个吸引点 | 依赖整个系统的理由 |
| --- | --- | --- |
| 练声/唱歌 | 五分钟开嗓、自动移调、歌曲分句 | 伴奏、音域、逐句证据、课程和弱句复练在同一项目 |
| 音调仪 | 打开即用、曲线稳、音域和录音完整 | 停止后得到更可靠的逐音/逐句分析和历史 |
| 视唱练耳 | EarMaster 级活动＋中文教学 | 错误能立即用钢琴、谱面、唱名和自己的歌曲复练 |
| 乐理/艺考 | 中文课程、题库、词典和多输入 | 知识点直接转成听、唱、弹、写的能力 |
| 钢琴 | 完美钢琴级键盘＋来音级学谱 | 一份谱可制、可弹、可唱、可听写、可伴奏，错误可定位 |
| 伴奏 | 分离、变调、循环、多轨和导出 | 处理完成无需换 App 就能练唱、练琴或生成谱面 |
| 制谱/识谱 | 拍照/PDF/MIDI/手工快速成谱 | 确认后立即播放、练习、分析、同步和教师批注 |

建议公开表达：

- “一份谱，能写、能听、能唱、能弹、能复练”；
- “一首歌，从伴奏处理到真正练会”；
- “钢琴、视唱练耳、乐理、练声、制谱和伴奏共享同一个学习项目”；
- “自动识别先给你检查，不把猜测当正确答案”；
- “本地即时能力与云端高质量能力协同，上传前由你确认”；
- “错误定位到音符、小节和句子，不只给一个总百分比”。

用户依赖应来自协调效率和长期价值，而不是封闭格式或无法导出的数据。用户必须能导入自有材料、导出标准格式、删除本地与云端数据，并看到处理位置和版本。

## 15. 恢复开发时的第一优先级

恢复条件仍是产品所有者明确要求恢复运行时开发。恢复后先读取远端最新 main、`AGENTS.md`、正式 DoD、状态矩阵、专业竞争路线、最新交接和本报告，再核对 P112 合并事实与 P113 起点，避免从过期 P106 假设重启。

建议的工程顺序：

1. 冻结 `ScoreDocument`、`PracticeTarget`、标准 note event/PCM、`AnalysisEvidence`、资源包与云中立 provider 契约；
2. 继续完成已经真实进行中的音频/音准切片，不因路线扩张丢弃已完成工作；
3. 钢琴用“完美钢琴演奏广度＋来音钢琴学习闭环”作为同一 DoD；
4. 启动 S1/S2，先做可编辑、可保存、可播放的乐谱基础，而不是先放一个拍照按钮；
5. 接入 MusicXML/MXL/MIDI round-trip，再做本地/云端 OMR 草稿和逐符号修正；
6. 把确认乐谱接入钢琴、视唱、节奏和练声，完成第一条乐谱黄金闭环；
7. 伴奏 A1/A2 与钢琴/音准接口冻结后并行，随后完成歌曲黄金闭环；
8. 补齐 EarMaster 级活动、来音乐理中文课程、统计和解释性自适应；
9. 服务条件成熟后接入中国区合规的同步、内容、重型算法和教师 provider；
10. 八条竞品任务、三档 Android、数据集、教师和目标用户证据汇合后，才定义真正的全能最终候选。

## 17. 最终决策摘要

1. 目标清单正式改为八款，练耳大师移出；
2. 钢琴不在完美钢琴和来音钢琴之间二选一：前者定义移动演奏器广度，后者定义学谱与多输入闭环，我方同时达到；
3. 来音制谱把原始 OMR 方向提升为完整乐谱工作台；识别只是输入之一，编辑、确认、播放、标准格式和全系统练习复用同等重要；
4. 安装包大小不作为砍功能理由，但资源仍要版本化、共享、可校验、可更新和可删除；
5. 最终产品同时建设本地和云端，不把当前本地阶段误当终点；
6. 每个模块先达到单项竞争力，再用统一乐谱、歌曲项目、分析证据和学习画像形成八款单项产品难以复制的整体优势；
7. 社区、公开谱库、排行榜、多人和商业外围与当前 canonical 排除冲突，必须另行显式决策，不能暗中计入或遗漏；
8. 当前报告只完成规划与静态证据，不宣称任何新运行时能力已实现，也不宣称准确度或体验已经超过竞品。
