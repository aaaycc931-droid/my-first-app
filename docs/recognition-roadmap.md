# 真实五线谱识别技术路线

## 当前 MVP 状态

当前应用已经完成了五线谱图片识别体验的 MVP 闭环，但识别结果仍然是模拟数据，尚未接入真实 OMR（Optical Music Recognition，光学乐谱识别）能力。

已支持能力：

- 已支持上传五线谱图片。
- 已支持上传后图片预览。
- 已支持 `/api/recognize` 接口，用于承接前端识别请求。
- 当前 `/api/recognize` 仍返回模拟识别结果，主要用于验证产品流程和前端展示。
- 已支持音高、时值、置信度、小节、拍点展示。
- 已支持 BPM 控制和钢琴播放，可基于识别结果结构进行演奏。

## 真实识别目标

真实五线谱识别的目标，是从用户上传的乐谱图片中提取可播放、可展示、可验证的结构化音符数据。核心能力包括：

1. **图片预处理**
   - 图片格式校验与标准化。
   - 灰度化、二值化、去噪、倾斜校正。
   - 提升五线谱线和音符符号的可检测性。

2. **五线谱线检测**
   - 检测每组五线谱的位置。
   - 识别谱表区域、行距、线距和系统边界。
   - 为后续音高判断提供空间基准。

3. **音符检测**
   - 检测符头、符干、符尾、连音线、休止符等音乐符号。
   - 区分单音、和弦、附点音符、休止符等不同元素。
   - 记录每个音符在图片中的位置和所属谱表。

4. **音高判断**
   - 根据符头相对五线谱线的位置判断音高。
   - 结合谱号、调号、升降记号还原实际 pitch。
   - 输出前端播放系统可使用的音高值。

5. **节奏判断**
   - 根据符头、符干、符尾、附点、连音等信息判断时值。
   - 结合拍号和小节线推算 measure 与 beat。
   - 输出可用于播放排序和节拍定位的 duration、measure、beat。

6. **输出标准识别结果数据**
   - 将识别结果统一为当前前端需要的数据结构。
   - 每个音符至少包含 pitch、duration、confidence、measure、beat。
   - 保持接口输出稳定，方便前端展示和播放复用。

## 推荐技术路线

### 第一阶段：使用成熟 OMR 工具转换为 MusicXML

MVP 阶段优先接入成熟 OMR 工具，而不是直接自研完整识别算法。推荐先调研并接入 Audiveris 或其他可用 OMR 工具，将用户上传的五线谱图片转换为 MusicXML。

这一阶段的目标：

- 接收前端上传的图片。
- 调用 Audiveris 或其他 OMR 工具进行识别。
- 生成 MusicXML 文件或 MusicXML 字符串。
- 保留识别过程中的错误信息，方便前端提示和后续调试。

### 第二阶段：解析 MusicXML

在拿到 MusicXML 后，后端需要解析其中的乐谱结构，并提取当前产品最需要的字段。

重点提取内容：

- `note`：音符或休止符信息。
- `duration`：音符时值。
- `measure`：小节编号。
- `beat`：小节内拍点位置。
- pitch 相关字段：step、octave、alter 等。
- 其他可选字段：voice、staff、confidence、tie、chord 等。

这一阶段不需要一次性支持所有 MusicXML 特性，应优先覆盖 MVP 播放需要的单声部或简单乐谱场景。

### 第三阶段：转换为当前前端需要的识别结果结构

解析 MusicXML 后，需要将结果转换为当前前端已经支持的识别结果结构，避免改动现有页面和播放逻辑。

建议输出结构保持稳定，例如：

```json
{
  "notes": [
    {
      "pitch": "C4",
      "duration": 1,
      "confidence": 0.92,
      "measure": 1,
      "beat": 1
    }
  ]
}
```

转换重点：

- 将 MusicXML 的 pitch 字段转换为前端可播放的音高字符串。
- 将 MusicXML 的 duration 转换为当前播放系统使用的时值单位。
- 根据小节和时值计算 beat。
- 如果 OMR 工具没有直接提供 confidence，可先使用默认置信度或基于识别状态生成粗略置信度。

### 第四阶段：用现有播放系统演奏结果

当前 MVP 已经具备 BPM 控制和钢琴播放能力，因此真实识别接入后，应继续复用现有播放系统。

这一阶段的目标：

- 前端仍调用 `/api/recognize`。
- `/api/recognize` 返回真实识别后的标准结构。
- 页面继续展示音高、时值、置信度、小节、拍点。
- 播放器根据真实识别结果和用户设置的 BPM 进行演奏。

## 为什么暂时不直接训练 AI 模型

MVP 阶段不建议直接训练自研 AI 五线谱识别模型，主要原因如下：

- **数据集要求高**：五线谱识别需要覆盖不同印刷质量、拍摄角度、清晰度、谱号、调号、拍号、音符组合和版式的数据。
- **标注成本高**：不仅要标注音符位置，还要标注音高、时值、小节、拍点、连音、休止符等结构化音乐信息。
- **准确率验证复杂**：真实可用的 OMR 不只看单个符号识别率，还要验证 MusicXML 结构、节奏一致性、播放效果和边界场景。
- **MVP 阶段应优先使用成熟 OMR 工具**：成熟工具可以更快验证产品闭环，把开发重点放在上传、预览、识别结果展示、播放和用户反馈上。

因此，当前更合适的路线是先基于 Audiveris 或其他成熟 OMR 工具搭建真实识别链路，等产品需求和样本数据稳定后，再评估是否训练自研模型或混合模型。

## 后续可能接入 Python/FastAPI 识别服务

后续可以将真实识别能力拆分为独立的 Python/FastAPI 服务，让 Next.js 应用专注于页面、接口编排和播放体验。

可能的服务架构：

1. Next.js 前端上传图片到 `/api/recognize`。
2. Next.js API 将图片转发给 Python/FastAPI 识别服务。
3. Python/FastAPI 服务负责：
   - 图片预处理。
   - 调用 Audiveris 或其他 OMR 工具。
   - 解析 MusicXML。
   - 转换为标准识别结果结构。
4. Next.js API 将标准结果返回给前端。
5. 前端继续使用现有结果展示和钢琴播放系统。

这样可以保持当前页面不受影响，同时为后续引入 OpenCV、MusicXML 解析库、OMR 工具调用、自研模型实验预留空间。

## MVP 接入原则

- 不修改当前页面交互，先保持上传、预览、识别、展示、播放流程稳定。
- 不改变前端依赖的识别结果结构，优先在后端做适配。
- 每次只接入一个能力，例如先打通 MusicXML 解析，再替换模拟结果。
- 优先保证可运行，再逐步提升识别准确率和边界场景覆盖。
